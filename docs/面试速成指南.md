## 一、用大白话解释你做的是什么

**一句话版**：我做了一个"矿业情报助手"，你问它"Pilbara 锂矿最近怎么样？"，它会自动查矿山数据、查价格、查新闻，然后给你一份带表格的结构化报告。

**面试官版**：这个项目基于 MCP 协议，把矿业数据拆成了三个独立的微服务（矿山生产、矿价行情、矿业新闻），然后通过一个 ReAct Agent 自动编排工具调用，最后合成结构化的 Markdown 情报简报。

## 二、你必须能解释的 5 个概念

### 1. MCP 是什么？

**简单版**：MCP 就像 USB 接口——任何 AI 应用只要支持 MCP，就能"插上"任何 MCP 工具。不用每次定制开发。

**面试版**：MCP（Model Context Protocol）是 Anthropic 发布的开放协议，用于 LLM 和外部工具之间的标准化通信。运行在 JSON-RPC over stdio/SSE 之上。核心概念是 Server 暴露 Tools/Resources/Prompts，Client 发现并调用。

### 2. 为什么用 3 个 Server 而不是 1 个？

**简单版**：矿山数据、价格数据、新闻数据来源不同。拆开的好处是：每个可以独立更新、独立部署、独立接入不同的 AI 应用。

**面试版**：遵循单一职责原则。松耦合带来的好处：(1) 单个 Server 故障不影响其他 Server；(2) 可以独立扩展，比如价格 Server 换成实时 API；(3) Claude Desktop 可以按需接入部分 Server。

### 3. ReAct 循环是什么？

**简单版**：AI 不是一次问完所有问题——先思考"我需要什么数据"，然后调用工具获取，看到结果后再决定是否需要更多数据，反复几轮后合成答案。

**面试版**：ReAct = Reasoning + Acting。每个迭代：LLM 输出 thought（思考）→ action（调用工具）→ observation（观察结果），直到 LLM 判断信息充分后生成最终回答。我这个实现最大 10 轮迭代。

### 4. MCP 的 stdio 和 SSE 有什么区别？

**简单版**：stdio 是"命令行管道"，SSE 是"网络连接"。我的项目用 stdio——不需要配网络端口，本地直接跑。

**面试版**：stdlib transport 通过子进程标准输入输出来通信——Agent 启动 MCP Server 为子进程，通过 JSON-RPC 消息交互。好处是零网络配置，适合本地工具。SSE transport 走 HTTP，适合远程服务。两种 transport 切换不需要改动 tool 定义。

### 5. 样本数据能换成真的吗？

**简单版**：能。data_loader 是抽象层——函数签名不变，内部把读 JSON 文件改成调 API 就行。

**面试版**：data_loader 模块做了接口抽象。`get_mine_by_name(name) -> dict` 这样的函数签名不依赖数据源。切换实时 API 只需要改 data_loader 的内部实现，Server 的 @mcp.tool() 和 Agent 的编排逻辑完全不动。

## 三、面试官可能问的 10 个问题（带答案）

### Q1: 为什么手写 ReAct 而不用 LangGraph？
> "我想展示对 MCP 协议和 LLM tool-use 机制的底层理解。手写 ReAct 循环只有 120 行，说明了整个流程：发送系统提示 → 获取 LLM 响应 → 处理 tool_use → 回传 tool_result → 循环直到 end_turn。用 LangGraph 两行代码就做完了，但面试官看不到我对协议的理解。不过我也准备了 LangGraph 版本作为对比——两个版本共享同一个 MCPClientManager，差别只在编排层。"

### Q2: 如果 LLM 选错了工具怎么办？
> "做了三层保护：(1) System prompt 明确写了推荐的工作流顺序；(2) 每个 tool 的 description 写了详细的参数和返回值说明；(3) 如果调用出错，Agent 会把错误信息作为 observation 传回，LLM 会自动尝试替代方案。实际测试中 Claude Sonnet 4 的工具选择准确率非常高。"

### Q3: 并发调用怎么处理？
> "当前实现是串行的——每轮迭代中 LLM 决定调用哪些工具，依次执行。如果工具之间没有依赖关系，可以改为并行调用——只需要把多个 tool_use block 的调用改为 asyncio.gather()。MCP ClientManager 已经支持这点，每个 session 是独立的。"

### Q4: MCP Server 崩溃了怎么办？
> "MCPClientManager 在启动时对每个 Server 做异常捕获——单个 Server 失败不影响其他 Server 启动。Agent 的 ReAct 循环天然支持'工具不可用则尝试替代方案'——如果调某个工具返回 error，LLM 会调整策略用其他可用工具。"

### Q5: 为什么数据存在 JSON 而不是数据库里？
> "因为这是 24h 限时项目，样本数据量小（7 个矿山 + 30 个价格点 + 15 条新闻），JSON 文件零依赖、易版本管理、易审查。如果是生产环境，矿山数据和价格数据应该入 SQL 数据库，新闻数据应该用向量数据库做语义检索。但 data_loader 的接口设计使得切换存储后端不影响上层。"

### Q6: 怎么保证 Agent 输出的报告质量？
> "做了几件事：(1) System prompt 强制规定了报告结构（摘要→产量→价格→投资信号→数据源）；(2) 每轮迭代 LLM 能看到之前的工具调用结果，判断信息是否充分；(3) 最大迭代数防止无限循环；(4) 每条数据标注来源，避免幻觉。如果要量化评估，可以用 RAGAS 的 faithfulness 和 answer relevancy 指标。"

### Q7: 为什么选 Anthropic SDK 而不是 OpenAI？
> "因为 Claude 的 tool-use API 设计跟 MCP 协议最契合——tool_use 和 tool_result 的消息格式天然支持多轮交互。而且 mcp-config.json 可以直接接 Claude Desktop 验证——面试官可以在自己电脑上复现。"

### Q8: Docker 的作用是什么？
> "解决环境问题。Docker 把 Python 版本、依赖包、文件路径全部封装好——面试官不需要配置 Python 环境，docker compose up 一键就能跑。容器内部 Agent 启动 3 个 MCP Server 为子进程，stdio 通信在同一个容器内原生运行。"

### Q9: 你知道 FastMCP 和底层 MCP SDK 的区别吗？
> "FastMCP 是 MCP SDK 的高层封装——用 @mcp.tool() 装饰器直接注册函数为 MCP tool，省去了手写 JSON-RPC handler。底层 SDK 需要手动处理 initialize、list_tools、call_tool 等协议消息。项目中 Server 用 FastMCP（快速开发），Client 用底层 SDK（展示对协议的理解）。"

### Q10: 这个项目最大的技术挑战是什么？
> "两个。第一是 MCP 的 stdio transport 在 Windows 下的异步上下文管理——anyio 的 cancel scope 必须在同一个 task 中进出，我用了 AsyncExitStack 来正确管理生命周期。第二是三个 Server 的 tool 函数命名冲突——Python 中 @mcp.tool() 装饰的函数会覆盖同名的导入函数，导致递归调用。我用 import as 别名解决了。"
